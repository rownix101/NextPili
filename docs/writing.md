# Documentation Style Pathway

从这里开始，借助统一的信息架构、中文用语与 **Example-first** 写法，写出可扫读、可对照、可落地的实现文档。

> 状态：已定 v1.4  
> 适用范围：`docs/**` 与根 `README.md` 中的说明性内容

读者扫一眼就知道「这是什么、长什么样、我照着做什么」——不是读一篇只讲原则的散文。

---

## 概览

本文规定 NextPili 文档怎么写。你可以用它：

- 建立全局图（Essentials）
- 跟 Guide 落地模块（Guides）
- 查路径与响应形状（Reference）
- 对齐外观与交互（Human Interface）

| 承诺 | 含义 |
|------|------|
| **目的先行** | 标题下第一句写清用途或结果，不写历史 |
| **用语像产品文档** | 冷静、第二人称、动词带结果；忌口语填充 |
| **Example-first** | 规则、流程、契约**必须带可对照的例子**；禁止只写「应该如何」 |
| **分层披露** | 概览建立模型；细节与表在 Topics / Reference |
| **一处定义** | 细节只在一处；别处链接，不复制后漂移 |
| **渐进路径** | 索引与 Guide 按「基础 → 构建 → 深入」组织阅读，而不是平铺标题 |

> **Important:** 没有例子的步骤、约束或 API 说明视为未完成。  
> 读者应能把文中 example 与真实抓包、崩溃日志、代码形状**对照**，而不是自行脑补。

文档的辨识度来自**句式**与**路径感**，不是来自 Markdown 装饰。  
索引页与长 Guide 的可读骨架：**概览建立语境 → 了解基础知识 → 构建 → 深入探索 → 探索更多**。正文用工程中文写定义与步骤。

## 了解基础知识

现在你已经知道本文要管什么，接下来掌握用语与句式——这比装饰标题更重要。

### 导语（Lede）

标题下第一句 = 能力或结果，常用 **动词并列** 或 **祈使 / 目的句**。

| | 示例 |
|--|------|
| 好 | 解析 B 站 `playurl` 响应，输出播放器可用的 `MediaSource`。 |
| 好 | 用 Cookie 槽位路由多账号请求，并在 POST 时附带 CSRF。 |
| 差 | 本章将介绍播放模块的设计思路与一些注意事项。 |
| 差 | 关于鉴权，其实有很多细节需要注意。 |

索引页 / Topics 条目下的**一行摘要**同样用「动词 + 结果」：

| 好 | 差 |
|----|----|
| 用 `VideoId` 解析 BV / av，失败时返回 `InvalidArgument`。 | 视频 ID 的一些说明。 |
| 从 [Architecture](./architecture.md) 建立分层，再读 domain 与 ffi。 | 架构相关内容。 |

### 中文气质

#### 定义句（Lede / 概览首段）

模式：**主语是什么 + 适用哪里 + 性质并列 + 结果**。

| | 句子 |
|--|------|
| 好 | `domain` 是无 IO 的业务模型层，适用于全部 crate 共享的概念。类型形状简明，错误语义集中；从设计上把协议细节挡在 `http` / `auth` 之外。 |
| 好 | 从这里开始，按分层文档与 Example-first 写出可对照的实现说明。 |
| 差 | `domain` 模块比较重要，里面会放很多东西，大家开发时要注意。 |

工程文档通常 **1–2 句定义 + 1 句结果**，不堆夸张修饰；保留结构：**是什么 → 性质 → 你能得到什么**。

#### 卡片 / Topics / 路径条目一行

模式：**疑问或主题名** + **一行动作摘要**。

| 好 |
|----|
| WBI 签名 — 为 Web 查询参数生成 `w_rid` / `wts`，降低 `-412`。 |
| 刚接触本仓库？ — 从 [Architecture](./architecture.md) 建立分层，再读 domain 与 ffi。 |
| 端点参数与响应形状的权威来源见 `docs/api`；以本文 + 实测为准。 |

#### 偏好词与句式

| 偏好 | 少用 / 不用 |
|------|-------------|
| 使用 X 来… / 借助… | 利用一下…、搞一下… |
| 进一步了解 / 详见 | 点击这里、点此查看 |
| 权威信息来源 / 以本文为准 | 大概以…为准吧 |
| 简明、安全（性质词克制使用） | 非常强大、超级方便、绝绝子 |
| 你可以从…开始 | 接下来我们先来聊聊 |
| 让你能… / 可… | 赋能用户去… |
| 操作指南与示例 | 一些文档和材料 |

第二人称「你」指实现者，或直接用祈使句省略主语。

### 概览常用句式

| 句式 | 作用 | 示例 |
|------|------|------|
| **X 是…，用来…** | 定义 | `domain` 是无 IO 的业务模型层，用来表达 ID、实体与错误语义。 |
| **使用 X 来…** | 用途 | 使用 WBI 为 Web 接口查询参数签名，降低 `-412` 拦截概率。 |
| **当…时，…** | 条件分支 | 当 `code == -101` 时，将会话标为未登录并触发重新登录流。 |
| **X 提供…** | 能力列举前 | `store` 提供账号密文、设备标识与非敏感设置的读写。 |
| **不要 / 禁止…** | 硬边界 | 禁止在 Flutter 侧持久化 `SESSDATA`。 |

概览里 1–3 段即可：定义 → 你用它做什么 → 边界或非目标。  
**不要**以「本文将介绍」「接下来我们讨论」开篇。

### 标题用词

| 类型 | 偏好 | 示例 |
|------|------|------|
| Guide 章节 | 动名词 / 动宾（做成某事） | 解析 `VideoId`；规范化 playurl；路由账号槽 |
| Pathway 式主节 | 短动作短语（读者阶段） | 了解基础知识；构建代码；深入探索 |
| 诊断 / 排查 | 祈使 | 确认是否为 CSRF 失败；检查 `buvid3` 是否缺失 |
| Reference 节 | 名词 | 参数；响应；错误码 |
| 索引分组 | 短名词 | Essentials；鉴权；端点 |

避免：`一些说明`、`杂项`、`其他`、`补充`（除非真是附录且有链接）。

### 人称与情态

| 用 | 不用 |
|----|------|
| 你 / 祈使（实现者视角）：「在 `core` 里把 `Error` 映射为 `AppError`。」 | 「我们其实可以考虑也许映射一下」 |
| 肯定契约：「必须」「禁止」「仅当」 | 「最好」「尽量」「一般说来」掩盖硬约束 |
| 未知就写清：「未实测；以抓包为准并回写本文。」 | 假装确定或含糊跳过 |
| 符号原名 + 反引号：`SESSDATA`、`AccountSlot` | 中英混名且不定义 |

软建议与硬约束分开写。硬约束进表或 **Important**；软建议用「可选」「若需要」。

### 段落节奏

| 节奏 | 适用 |
|------|------|
| 短判断 → **贴证据** → 一句解释或详见链接 | 排错、协议、错误映射 |
| **代码块 → 一句能力说明** → 再代码 → 再一句 | 教模型、API 形状 |
| **阶段导语 → 一小段语境 → 资源列表** | 索引、长 Guide 的阅读路径 |

代码后点题的固定模板：

```text
[最小 example]

用 `VideoId` 解析路由参数，得到 `Bvid` 或 `Aid`，再填入详情请求。
```

工程文档默认顺序：

1. 一句话说清现象或规则（可极短）  
2. **立刻**给 example  
3. **代码/表后的一句点题**（说明读者刚看到的东西能做什么）  
4. 例外、映射表、See Also  

一段一个意思；默认 1–3 句。超过三个并列事实 → 表或列表。  
**禁止**「先写半页原理，最后丢一个无说明代码块」。

### 禁止的填充语

| 删除或改写 | 原因 |
|------------|------|
| 简单来说 / 其实 / 众所周知 | 无信息 |
| 非常重要的是…（却不给例子） | 空强调 |
| 如下所示。（单独成段） | 应用一句话点题再贴代码 |
| 关于 X 的一些想法 | 不是工程文档标题 |
| 方便起见我们… | 含糊责任 |

## 构建页面

在动手写模块文档前，先认清文档体系与页面骨架。井然有序的文档提高可读性、保证一致性，并避免日后细节漂移。

### 文档体系（信息架构）

| 层级 | 角色 | 落点 | 读者问题 |
|------|------|------|----------|
| **Documentation Home** | 入口与地图 | [docs/README.md](./README.md) | 从哪读起？ |
| **Essentials** | 全局图、边界、ADR | [architecture.md](./architecture.md) | 系统如何拼起来？ |
| **Guides** | 可指导实现的设计 | [design/](./design/README.md) | 这个模块怎么做？ |
| **Human Interface** | 视觉、交互、动效、平台 | [ux/](./ux/README.md) | 长什么样、怎么操作？ |
| **Reference** | 可查契约 | [api/](./api/README.md) | 路径、参数、码表？ |
| **Style** | 文档规范 | 本文 | 怎么写文档？ |
| **Human Interface · Copy** | 产品界面用词 | [ux/copy.md](./ux/copy.md) | 按钮/错误怎么措辞？ |

**细节只在一处定义**；其它文档链接引用。

#### 边界（禁止串台）

| 内容 | 归属 | 不要写在 |
|------|------|----------|
| 分层、依赖、跨 crate ADR | `architecture.md` | 端点参数表、色值 token |
| 模块职责、状态、流程 | `design/*` | 复制整张 API 字段表 |
| 端点、签名、Cookie、响应 | `api/**` | 架构 ADR 长文 |
| 配色、玻璃、动效、快捷键 | `ux/*` | `design/` 展开 token 全集 |
| Flutter 目录 / Riverpod 落点 | `design/flutter.md` | UX 视觉细则 |

### 页面解剖

#### 标准 Guide 页

```markdown
# 短标题

> 状态：草案 | 已定 · vX.Y  
> 相关：[上级](…) · [平行主题](…) · [Reference](…)

一句话导语（动词 / 结果）。

---

## 概览

定义 + 用途 + 边界。不贴参数全集。

## Topics

### 主题 A

点题句。

#### Example

（输入 / 输出 / JSON / 表）

规则表或步骤。

### 主题 B

…

## See Also

- [相关 Guide](…)
- [相关 Reference](…)
```

长 Guide 或「从零跟到落地」类页面，可把 `Topics` 改成阶段名（不必硬套编号）：

```markdown
## 概览
## 了解基础知识
## 构建 …
## 深入探索
## See Also
```

每一阶段：**导语 1–3 句 → example 或步骤 → 可选资源列表**（链到 Reference / 平行 Guide）。

#### 索引页

索引 = **地图 + 阅读路径 + 边界**。  
每条链接一行摘要（动词 + 结果）。不在索引里展开实现。

阅读路径优先写成阶段列表（工程新人 / UI / 协议回写），见 [docs/README.md](./README.md)。

#### Reference 页

```markdown
# 端点或机制名

一句话用途。

## 请求

| 项 | 值 |
|----|-----|
| Method | GET |
| Path | `/x/…` |
| Auth | Cookie + WBI |

### 参数

| 名称 | 类型 | 必填 | 说明 |
|------|------|------|------|

## 响应

### Example · 成功

```json
{ "code": 0, "message": "0", "ttl": 1, "data": { } }
```

### 字段

| 字段 | 类型 | 说明 |

### Example · 失败（可选但推荐）

```json
{ "code": -101, "message": "账号未登录", "ttl": 1 }
```

## 备注

陷阱、风控、与其它端点差异（仍尽量给例子）。

## See Also
```

参数列序固定：**名称 → 类型 → 必填 → 说明**。

### 元数据与状态

```markdown
> 状态：草案 v0.3  
> 约束：无 IO  
> 相关：[architecture](../architecture.md) · [api](../api/README.md)
```

| 状态 | 含义 |
|------|------|
| **草案** | 可讨论；实现可不完全对齐 |
| **已定** | 实现与评审以本文为准；破坏性变更要在 PR 说明 |
| **废弃** | 文首标明替代文档 |

小修措辞不升版；边界 / token / 端点契约 / ADR 变更升版。

### 标题与文件

| 层级 | 用途 |
|------|------|
| `#` | 页标题，全文唯一 |
| `##` | 概览、主 Topic / 阶段、See Also |
| `###` | Topic 内小节 |
| `####` | 少用；Example 子节可用 |

文件：索引用 `README.md`；主题用 `kebab-case.md` 或既有短名。  
**禁止**无索引挂载的孤儿页；新页必须进入所属 README 地图。

## 用 Example 讲清楚

规则只有落到可对照物上，才算写完。这是本规范的强制要求。

### 规则

凡出现下列内容，**同一小节内**必须出现至少一种 example：

| 内容类型 | 必须有的 example 形态 |
|----------|----------------------|
| 步骤 / 算法 | 输入 → 输出的完整小例子，或逐步数值/字符串 |
| 数据模型 / DTO | 最小结构（Rust 形状、JSON、或 `text` 代数类型） |
| API 端点 | 最小请求要素表 + 成功 JSON（只留实现需要的字段） |
| 错误映射 | `code` / 日志原文 → 领域错误 对照表或前后片段 |
| 约束 / 禁止 | ✅ 合法例子 + ❌ 非法例子（可同表） |
| UI / Token | 具体 token 名与值，或短 Dart 片段；禁止只写「使用语义色」 |
| 配置 / 命令 | 可复制的命令或配置块 |

> **Note:** 概览可以只有定义句；**Topics 里每一条可执行规则**都要能落到 example。  
> 长文可在概览末用列表点出后面有哪些 worked example（可选）。

### 好例子的标准

| 标准 | 说明 |
|------|------|
| **可对照** | 读者能拿去和抓包、单测 fixture、崩溃日志比对 |
| **最小** | 只含讲清这一点所需字段；不贴整页响应 |
| **完整** | 对这一点自洽：该有的输入输出、成功/失败成对时成对给 |
| **有前后文** | 代码块前一句说明「你在看什么」；后可接「因此…」 |
| **标明身份** | 示例 / 反例 / 仅说明用伪代码，避免被当成可提交实现 |

### 反模式（只说怎么做）

**差（只有原则，无可对照物）：**

> 签名前需要对参数排序，然后做一轮字符替换，最后算 MD5。  
> 注意不要漏掉 `wts`，否则容易 -412。

**好（先 example，再步骤；块后可再补风险）：**

对查询参数做 WBI 签名：排序 → 过滤 → 拼接 mixin → MD5，得到 `w_rid`。

输入（已含业务参数，未签名）：

~~~text
foo=1&bar=2&wts=1700000000
~~~

排序后拼接（示意）：

~~~text
bar=2&foo=1&wts=1700000000{MIXIN_SUFFIX}
~~~

输出查询参数：

~~~text
bar=2&foo=1&wts=1700000000&w_rid=<32位hex>
~~~

用字典序与 mixin 把可变参数收成稳定字符串，再哈希为 `w_rid`。

| 步骤 | 动作 |
|------|------|
| 1 | 加入 `wts`（Unix 秒） |
| 2 | 按 key 字典序排序；过滤 `!'()*` |
| 3 | 拼接 mixin（由 `img_key`/`sub_key` 重排得到） |
| 4 | MD5 → `w_rid` |

> **Warning:** 缺少 `wts` / `w_rid` 或时钟偏差过大时，接口常返回 `-412`。

（真实文档写完整 mixin 规则与链接；此处只示范「先例子后点题再步骤」。）

### Example 版式

#### 主模板：代码 → 一句点题

能力说明**贴在例子后面**，写「这段东西让你能做什么」，不写「如下所示」。

~~~rust
pub enum VideoId {
    Bvid(String),
    Aid(i64),
}
~~~

用枚举表达 BV / av 两种标识，避免在调用处散落字符串约定。

可连续多组（每组一个焦点）：

~~~json
{ "code": 0, "message": "0", "ttl": 1, "data": { "isLogin": true } }
~~~

成功时业务载荷在 `data`；先判断 `code == 0` 再反序列化。

~~~json
{ "code": -101, "message": "账号未登录", "ttl": 1, "data": null }
~~~

将 `-101` 映射为 `Error::Unauthenticated`；`core` 再转为 FFI 可见的 `AppError`。

#### 辅模板：点题 → 证据

排错、对照日志时，允许**先一句判断再贴证据**：

未登录访问需登录接口时，响应形如：

~~~json
{
  "code": -101,
  "message": "账号未登录",
  "ttl": 1,
  "data": null
}
~~~

#### 成对与 Callout

| | Example |
|--|---------|
| ✅ | `VideoId` 解析 `"BV1xx411c7mD"` → `Bvid(...)` |
| ❌ | `VideoId` 解析 `""` → `Error::InvalidArgument` |

- **Note / Important / Warning** 点出规则或风险

## 极简全页示例（片段）

```markdown
# `VideoId` 解析

> 状态：已定  
> 相关：[domain](./domain.md) · [video 端点](../api/endpoints/video.md)

把用户输入或路由参数解析为 `Bvid` 或 `Aid`；无法识别时返回 `InvalidArgument`。

## 概览

播放与详情请求必须带 `bvid` 或 `aid` 之一。`VideoId` 集中解析规则，避免 UI 与 `http` 各写一套。

## 了解基础知识

### 合法输入

| 输入 | 结果 |
|------|------|
| `BV1xx411c7mD` | `Bvid("BV1xx411c7mD")` |
| `12345678` | `Aid(12345678)` |
| `av12345678` | `Aid(12345678)` |

### 非法输入

| 输入 | 结果 |
|------|------|
| `""` | `Error::InvalidArgument` |
| `BV` | `Error::InvalidArgument` |
| `av` | `Error::InvalidArgument` |

## 构建请求参数

```text
VideoId::Bvid("BV1xx…")  →  query bvid=BV1xx…
VideoId::Aid(12345678)   →  query aid=12345678
```

用解析结果直接填查询参数，调用处不必再判断前缀。

## See Also

- [playurl 规范化](./media.md)
```
